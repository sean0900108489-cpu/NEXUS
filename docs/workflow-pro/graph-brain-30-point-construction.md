# Graph Brain 30-Point Construction Notes

Generated during the V23 Workflow Pro construction pass.

## Goal

The next 30-point foundation target is to let the Graph area include a Brain chat/planning surface that can:

1. Understand a natural-language workflow request.
2. Plan the required node path, node order, and missing capabilities.
3. Produce strict `nexus.workflow.v1` JSON.
4. Append the generated workflow as a new group on the canvas.
5. Avoid mutating or rewriting existing workflow groups.

## Current Implementation

### Core Planner

File: `src/lib/workflow-pro/graph-brain-planner.ts`

This file introduces `nexus.workflowPro.graphBrainPlannerResult.v1`.

The planner is deliberately split into two conceptual stages:

- Intent Architect
  - Reads the operator request.
  - Reads the current RuntimeLite summary.
  - Chooses the best existing workflow draft shape.
  - Explains the node path and rationale.
  - Reports missing capabilities instead of pretending they exist.

- JSON Contract Compiler
  - Produces strict `nexus.workflow.v1` JSON.
  - Validates the wrapped Brain proposal through the existing proposal validator.
  - Marks the insertion mode as `new-workflow-group`.

The first implementation is deterministic so the product remains testable even when the LLM API is unavailable.

### Runtime Capability Report

File: `src/lib/workflow-pro/capability-inventory.ts`

Graph Brain now receives `architect.runtimeCapabilityReport`, a machine-readable summary of the real RuntimeLite execution layer:

- execution mode is `ready-parallel`
- workflow timeout policy is `none`
- pause control uses `abort-signal`
- fan-out and fan-in shapes are detected
- native ready-node parallel execution is explicitly `true`
- join node support is explicitly `false`
- topology validation status is included

This lets the Brain distinguish between fan-out branches that can already run in ready-node parallel batches and explicit join nodes that still need implementation.

RuntimeLite now executes ready nodes in parallel batches instead of awaiting every node one by one along the validated topological path.

The scheduler keeps the existing topology validator and still requires one connected `input.text` start node. During execution it repeatedly finds every pending node whose upstream packets already exist, then runs that ready set through `Promise.all`.

This upgrades fan-out workflows such as:

```text
input -> llm-a -> merge
input -> llm-b -> merge
```

so `llm-a` and `llm-b` can start in the same batch. Fan-in still uses the existing automatic ContextPacket merge behavior. A user-configurable explicit join node remains a future workflow node capability.

### Server Route

File: `src/app/api/workflow-pro/brain-draft/route.ts`

This route keeps API keys on the server. It accepts:

- `operatorRequest`
- optional `runtimeLite`
- optional `templateHint`
- optional `modelSettings`

If `OPENAI_API_KEY` exists, the route can ask the OpenAI Responses endpoint for an Architect overlay. The deterministic compiler still owns the final workflow JSON, so malformed model output cannot corrupt the canvas append path.

If the API call fails, the route returns the deterministic planner result.

### Brain Thread Memory

The panel now records recent Brain turns in a local thread. Each `Think` run adds:

- Operator Request
- Intent Architect
- JSON Contract Compiler
- Screen Apply Gate

The thread is intentionally local UI state for this phase. It gives the operator a visible chat-like handoff without adding database or sync complexity before the contract is stable. The server route accepts the latest thread messages as sanitized `conversation` context for the optional Architect overlay, but final workflow JSON is still generated by the deterministic compiler.

### Runtime Evidence

The Brain handoff now includes `nexus.workflowPro.runtimeEvidence.v1`.

This report reads the existing local `runtimeLite.runs` snapshot and summarizes:

- latest run status
- run duration
- node execution timeline
- failed/running/success counts
- output previews
- generated artifact IDs
- warnings when the canvas has no execution history yet

The Graph Brain panel displays this as a compact `Runtime Evidence` section. This is intentionally marked as a local workspace runtime snapshot, not durable backend trace persistence.

The right-side `Trace` panel also displays the same report as `Local Workflow Evidence`, separate from backend observability events. This gives the operator one place to inspect both local Workflow Runtime evidence and cloud trace events without confusing their sources.

The Trace panel can export the local workflow evidence as `nexus.workflowPro.runtimeEvidenceManifest.v1`. The exported JSON points to `runtimeLite.runs` as the current source of truth and marks backend persistence as the next gate instead of pretending local evidence is already durable server trace.

### Run History Groups

File: `src/lib/workflow-pro/run-history-groups.ts`

The Trace panel now includes `nexus.workflowPro.runHistoryGroups.v1`, a compact group-level view for canvases that contain many appended workflow groups at the same time.

The report:

- infers appended workflow group IDs from node IDs such as `wf_group_<id>_node_<name>`
- falls back to `workspace-root` for legacy or ungrouped nodes
- summarizes node count, run count, status counts, latest run, and artifact count per group
- renders in the right-side Trace panel under `Local Workflow Evidence`

This keeps the product aligned with the user rule that Brain-generated workflows append as new groups instead of mutating existing groups.

### Trace Sync Retry Control

Files:

- `src/store/nexus-store.ts`
- `src/store/nexus-store.test.ts`
- `src/components/nexus/nexus-ops.tsx`

The Trace panel Group Inspector now exposes a `Resync Trace` control for the selected group's latest run.

The action calls:

```text
retryWorkflowRuntimeTraceSync(runId)
```

The store action:

- finds the run inside the active workspace RuntimeLite snapshot
- refuses to retry missing, queued, or running runs
- marks the run as `traceSync.status = syncing`
- republishes the sanitized trace through `publishWorkflowRuntimeTrace`
- stores either `synced` evidence with backend event metadata or `failed` evidence with a retryable error
- queues workspace cloud sync after the retry result is recorded

The UI control is intentionally group-scoped through the inspector's latest run. If the selected group has no local run yet, the button is disabled. This makes failed trace persistence recoverable without letting the UI invent trace records.

The backend write boundary remains unchanged: `POST /api/v1/workflows/runtime-trace` requires editor access, rejects raw payload fields, and records denied writes as security audit evidence.

Files:

Workflow Pro file inputs now have a machine-readable compiler lane manifest:

```text
nexus.attachmentCompilerManifest.v1
```

The manifest defines independent lanes for:

- `text`
- `image`
- `audio`
- `video`
- `archive`
- `document`
- `binary-reference`

Every lane currently defaults to the no-op compiler unless text can safely pass through as text. Reserved lanes still record their future adapter slots, such as `speech-to-text`, `zip-expand`, `video-transcode`, and `pdf-text-extractor`.

Composer uploads now resolve the lane from MIME type before writing artifact metadata. File node contracts expose the same manifest to the Graph Brain and future LLM workflow planners. Runtime file node packets include a compact lane summary so execution evidence can show which compiler lane carried the attachment without copying raw payloads into `ContextPacket.metadata`.

This is a contract foundation, not a claim that advanced compilers already execute. Actual audio transcription, archive expansion, PDF extraction, and media transcoding remain future adapter implementations.

### Attachment Compiler Execution Pipeline

Files:

- `src/lib/attachments/attachment-compiler-execution.ts`
- `src/lib/attachments/attachment-compiler-execution.test.ts`
- `src/lib/workflow-runtime-lite/executors.ts`
- `src/lib/workflow-runtime-lite/runner.test.ts`

Workflow Pro file nodes now run through an executable compiler pipeline:

```text
nexus.attachmentCompilerExecution.v1
```

The first implementation is intentionally conservative:

- text-safe attachment references become `passthrough` compiler results
- image, audio, video, archive, document, and binary inputs remain `reference-only`
- future adapter slots stay visible on each result
- RuntimeLite packets keep a compact execution summary instead of nested raw result objects

The packet boundary is designed around the existing `ContextPacket.metadata` sanitizer. Full compile results are tested at the attachment compiler layer, while runtime packets store compact `resultSummary` strings such as:

```text
source.zip:archive:reference-only:artifact-raw
```

This keeps the graph runtime safe from large payloads and deep metadata while still proving that every file node execution crosses a compiler boundary before downstream nodes receive the packet.

### Durable Workflow Group Records

Files:

- `src/lib/backend/observability/workflow-group-records.ts`
- `src/lib/backend/observability/workflow-group-records.test.ts`
- `src/app/api/v1/workflows/groups/route.ts`
- `src/app/api/v1/workflows/groups/route.test.ts`
- `src/lib/workflow-pro/group-record-client.ts`
- `src/lib/workflow-pro/group-record-client.test.ts`
- `src/store/nexus-store.ts`
- `src/store/nexus-store.test.ts`

Workflow Pro groups now have a narrow durable record path:

```text
Graph Brain append
-> appendWorkflowRuntimeGroup
-> publishWorkflowGroupRecord
-> POST /api/v1/workflows/groups
-> workflow.group_record.upserted system event
```

The durable record uses:

```text
nexus.workflowPro.groupRecord.v1
```

The route stores group-level metadata in `system_events` through ObservabilityService. It records group ID, group label, node count, edge count, node type counts, contract schema, compiler manifest schema, and capability gaps.

The route rejects raw keys such as `rawText`, `displayText`, `inputSnapshot`, `outputSnapshot`, `prompt`, `dataUrl`, `binary`, `apiKey`, `providerKey`, `token`, and `secret`.

Editor or higher is required for writes. Viewer attempts are denied and leave security evidence, matching the existing runtime trace write boundary.

The store publishes group records in the background after a workflow group is appended. A group record publish failure does not block the canvas append, because workflow design should not fail just because durable observability is temporarily unavailable.

### Durable Group Record Reader

Files:

- `src/lib/workflow-pro/durable-group-records.ts`
- `src/lib/workflow-pro/durable-group-records.test.ts`
- `src/components/nexus/nexus-ops.tsx`

The Trace panel Group Inspector now includes:

```text
nexus.workflowPro.durableGroupRecordReport.v1
```

The reader compares the selected workflow group against loaded workspace observability events. It looks for:

```text
eventType = workflow.group_record.upserted
metadata.workflowGroupId = <selected group id>
```

It reports explicit states:

- `no-group`
- `not-loaded`
- `matched`
- `missing`

This separates three important realities that previously looked identical on screen:

- no workflow group is selected
- workspace events have not been refreshed yet
- workspace events were loaded, but no durable group record matched

The first UI implementation does not add a new backend query route. It reads the workspace events already loaded by the Trace panel's `Refresh` action and renders the result inside the selected group's inspector card. This keeps the surface conservative while still making durable group records visible to the operator.

### Workspace Permission Matrix

Files:

- `src/app/api/v1/observability/observability-route-helpers.ts`
- `src/app/api/v1/observability/metrics/route.ts`
- `src/app/api/v1/observability/observability-route-helpers.test.ts`
- `docs/workflow-pro/workflow-permission-matrix.md`

Observability read access now matches the action name. Read-like workspace observability actions require `viewer`, while write-like observability actions such as future workflow trace writes require `editor`.

Current route boundary:

- events read: viewer or higher
- trace projection read: viewer or higher
- usage metrics read: editor or higher
- future workflow trace write: editor or higher

This prevents the account-specific failure mode where a privileged account can inspect or generate workflow state while a valid lower-role workspace member is incorrectly blocked from read-only evidence.

## Supported Baseline Requests

### Image/File Input To Two LLMs

Natural language shape:

```text
我把圖填在 input 傳上之後，它可以連接兩個不同我已經設定好提示詞的 LLM 後給我最終答案。
```

Expected workflow shape:

```text
input.text -> node.file -> model.llm -> model.llm -> output.text
```

### Audio Prompt To Image Reverse Fan-Out

Natural language shape:

```text
我想傳一個語音提示詞之後生成圖像，再從那個圖像反推出提示詞，讓另外三個 LLM 更改風格後經過圖片模型生成最終答案。
```

Expected workflow shape:

```text
input.text -> node.file -> model.llm -> model.image -> model.llm
model.llm -> model.image -> output.text
model.llm -> model.image -> output.text
model.llm -> model.image -> output.text
```

The current RuntimeLite layer executes fan-out branches as ready-node parallel batches. The Brain must still report explicit join nodes and advanced media understanding as future capabilities.

## Safety Rules

- Do not put `OPENAI_API_KEY` in client components.
- Do not let model output directly mutate the graph.
- Do not replace the current canvas when the user asked for a new workflow group.
- Do not hide missing capabilities.
- Keep the deterministic compiler path available for tests, offline use, and malformed model responses.
