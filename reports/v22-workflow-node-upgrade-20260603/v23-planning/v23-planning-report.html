01

起點：這不是「再做一個聊天功能」，而是讓 NEXUS 有可被理解的 workflow 語言。

V23 的核心價值不是把 LLM 接到更多按鈕，而是讓整個 workflow 可以被一份 JSON 描述、驗證、生成、執行、反省、優化。

本次修正後的任務定義

你的「大腦」不是一個事後看 logs 的監控器。它應該在 workflow 開始之前就知道整條線在做什麼：誰是 input、哪幾個 LLM 協作、哪個節點負責圖片或影片、哪些節點並行、哪些節點串行、資料如何通過 edge、每個節點為什麼存在，以及如果要優化，應該改 prompt、改模型、加控制閥、加 compiler，還是請 Codex 開發新能力。

所以 V23 不應先從 UI 按鈕或資料表開始。第一個穩定起點是 canonical workflow contract：一份 `nexus.workflow.v1` JSON，它能被人看懂、被 LLM 接手、被 validator 檢查、被 graph UI 渲染、被 runtimeLite materialize，也能被後端版本化保存。

一句話 Goal

讓 NEXUS 從「能畫節點、能跑模型」升級成「能用一份 JSON 生成和理解整個智能工作流，並由 workflow brain 提出可執行升級方案」。

contract-first brain-readable validator-gated runtime-bridged
02

為什麼上一版看起來模糊：它講對方向，但沒有把「怎麼抵達」拆成可讀頁面。

問題不是方向錯，而是資訊壓縮太狠，缺少中間的工程層、資料層、UI 層、驗證層與收斂層。

缺少問題拆解

上一版說要做 contract，但沒有清楚展示目前 export、backend snapshot、runtimeLite、artifact history 各自的責任，因此你會難以判斷到底要改哪裡。

缺少欄位營養

只列欄位名稱不夠。你需要知道每個欄位為什麼存在、誰會讀它、誰能寫它、錯了會造成什麼風險，以及第一版能不能先簡化。

缺少落地節奏

如果沒有從 planning 到 source folder、validator、bridge、UI、後端、測試的順序，報告就像理念宣言，不像可以交給 Codex 執行的計畫。

本版修正策略

每頁只回答一個高價值問題，並在最後五頁逐步收斂：研究結論 → 檔案地圖 → 實作輪次 → 風險門檻 → 第一輪工程入口。

03

V22 現況基線：已經有可以承接 V23 的 runtime、graph 與資產紀錄。

V23 不是重做，正確做法是加一層設計合約，讓既有系統變得可解釋、可匯入、可驗證。

既有層 目前用途 V23 應怎麼接
Workspace Snapshot 完整保存 agents、panels、settings、graph、runtimeLite、notebooks。 保留為備份/還原格式，不改成大腦合約。
Backend Cloud State 保存 cloud snapshot，投影 graph/settings/agent/memory/tool_state/branch。 後續新增 workflow contract version，而不是塞進現有 projection。
Workflow Runtime Lite 執行 input/model/output nodes、edges、runs、ContextPacket。 透過 `to-runtime-lite` 橋接已驗證 contract。
Artifact Vault 保存生成結果、metadata、下載入口、歷史紀錄。 成為 image/video/audio/tool output 的標準 artifact policy。

基線判斷

目前不缺「能不能跑」的基礎。缺的是「這份 workflow 到底為什麼這樣設計」的語義層。這就是 `nexus.workflow.v1` 的位置。

04

Workflow Brain 的正確定義：先讀懂設計，再看證據，最後提出可審核改版。

這個大腦不是 runtime owner，而是 workflow design intelligence。

它必須先知道的東西

  • Intent：整個 workflow 為什麼存在，要幫使用者完成什麼。
  • Topology：哪些節點串行、哪些並行、哪些是分支、哪裡是匯流。
  • Node purpose：每個節點的角色、輸入、輸出、限制、模型設定、prompt 目的。
  • Edge reason：每條線傳什麼、為什麼這樣傳、packet schema 是什麼。
  • Capability boundary：目前平台能做什麼、不能做什麼、哪些只是計畫。

它不能直接做的事

不能繞過 validator 直接改 runtime。不能假裝不存在的功能已經存在。不能自動套用改版而不產生 diff。不能把 service role 或 API key 交給前端。不能把每一次想法都變成資料庫 migration。

Brain read order: 1. canonical workflow contract 2. capability inventory 3. runtime evidence 4. artifact history 5. user goal delta Brain write order: 1. analysis 2. optimizedWorkflow draft 3. diff summary 4. missingCapabilities 5. suggestedCodexTasks
05

設計原則：Contract first、runtime bridged、backend versioned、UI explainable。

這四條原則會防止 V23 變成功能堆疊或架構混亂。

Contract First

任何 workflow 都先被表示成 canonical JSON。Graph UI 是它的視覺表現，runtimeLite 是它的執行表現，backend 是它的持久化表現。

Runtime Bridged

runtimeLite 不需要背負所有語義。它只吃 validator 通過後的 materialized execution plan，保留現在穩定的執行職責。

Backend Versioned

後端不是只存最新狀態，而要能區分 draft、published、proposal、run evidence、artifact record。

UI Explainable

Graph 不只顯示節點，還要顯示 validation、capability、reason、rationale、artifact、proposal。

Validator Gated

匯入、執行、發布、brain proposal apply 都必須經過 validator，不讓 LLM 的好主意直接污染 runtime。

Human Reviewable

大腦可以產生新版 workflow，但第一版不做自動套用。先產生 diff 和風險，再由人或 Codex 批准。

06

三層真相：Snapshot、Contract、Runtime 必須分開。

這是整個架構最重要的分水嶺。混在一起，後續每個功能都會變難。

層問的問題主要讀者
Snapshot如何完整恢復使用者工作區？同步系統、備份、回復流程
Contract這個 workflow 設計上想做什麼？人、LLM、validator、Graph UI
Runtime這次執行到哪裡，產生了什麼？executor、run monitor、artifact system

決策

不要把左上角 Export 改成 contract。正確做法是新增「Export Workflow Contract」或在 export 裡包含 `workflowContracts[]`，但仍保留完整 workspace snapshot。

WorkspaceSnapshot contains UI + state + runtime backup nexus.workflow.v1 contains design intent + topology + capabilities + validation target RuntimeLite contains execution nodes + run state + packets + abort/pause evidence
07

`nexus.workflow.v1` 的目標：一份 JSON 同時服務人、LLM、UI、validator、runtime。

它不是越大越好，而是剛好足以描述工作流的設計理由與執行邊界。

給人看

能看懂 workflow 的目的、節點角色、順序、限制、成果、風險與為什麼這樣連。

給 LLM 看

能直接接手設計、批判、改寫，不需要猜 UI 裡每個節點的語義。

給 UI 看

Graph UI 可以從 contract 渲染節點、handles、validation badges、inspector 欄位。

給 validator 看

能檢查 schema、id、edge、handle、capability、compiler、artifact、security。

給 runtime 看

透過 bridge 轉成 runtimeLite nodes/edges/startNodes/parallelGroups。

給後端看

能版本化保存、比較 diff、追蹤 proposal、連到 run evidence 與 artifacts。

08

Schema 核心欄位：第一版要剛好，不能貪大。

V1 的成功不是覆蓋所有未來，而是建立穩定骨架，讓未來功能可插入。

{ "schema": "nexus.workflow.v1", "workflow": { "id": "wf_y2k_storyboard", "name": "Y2K Fashion Board Generator", "version": "0.1.0", "intent": "Generate a 16:9 fashion concept board from a prompt.", "successCriteria": ["image generated", "artifact saved", "download available"] }, "capabilityInventory": {}, "nodes": [], "edges": [], "execution": {}, "artifactPolicy": {}, "brainContract": {} }

workflow

描述目的、版本、成功標準，讓人和大腦知道這份流程為什麼存在。

capabilityInventory

告訴大腦目前平台可用與不可用的能力，防止憑空設計。

nodes/edges

描述工作流拓撲，但每個節點和連線都要有 purpose / reason。

09

Node 定義：節點不是卡片，它是可驗證的工作單元。

每個節點都要讓大腦知道角色、設定、限制、輸入輸出與失敗時怎麼處理。

{ "id": "image_model_1", "type": "model.image", "label": "Image Model", "purpose": "Generate a fashion board from refined prompt.", "rationale": "Image model is isolated so quality/aspect can be tuned.", "config": { "provider": "openai", "model": "gpt-image-2", "quality": "standard", "aspectRatio": "16:9" }, "io": { "inputs": [{ "handle": "prompt", "packet": "text.prompt" }], "outputs": [{ "handle": "image", "packet": "artifact.image" }] }, "limits": ["requires image API key", "produces artifact only after storage"] }

節點的最低營養

  • type 決定 validator 和 renderer。
  • purpose 決定人和大腦是否理解。
  • config 決定 runtime materialization。
  • io 決定 edge 是否接得起來。
  • limits 決定大腦不能亂承諾。
10

Edge 與 Packet：線不是線，是資料合約。

如果 edge 沒有 packet contract，workflow brain 就只能猜資料如何流動。

source / target

描述哪個節點的哪個 handle 連到哪個節點的哪個 handle。這會直接對應 React Flow handles 和 runtime edge。

mode

描述 serial、parallel、conditional、merge、loop-safe 等資料流模式。第一版可以只支援 serial/parallel/merge。

reason

描述為什麼要把這個 output 交給下一個 input。這是大腦能提出優化的關鍵。

{ "id": "edge_prompt_to_image", "source": "prompt_refiner", "sourceHandle": "refinedPrompt", "target": "image_model_1", "targetHandle": "prompt", "mode": "serial", "reason": "The image model should receive the refined visual prompt, not raw chat text.", "packetContract": { "kind": "text.prompt", "required": ["text", "language", "styleHints"], "optional": ["negativePrompt", "referenceAssetIds"] } }
11

拓撲：要能描述串聯、並聯、分支、匯流，但第一版要控制範圍。

你要的大腦知道 workflow 順序，這件事必須被寫進 contract。

拓撲型態V23 V1 狀態原因
Serial必做input → prompt → model → output 是基礎路徑。
Parallel必做多模型、多 prompt、多資產生成需要並行。
Merge必做多輸出需要匯總給最後節點或大腦。
Conditional先設計、後實作if/else 控制閥重要，但要先有 packet schema。
Loop延後任意 loop 容易造成執行與成本風險。

第一版執行排序

`execution.order` 不必成為唯一真相，但要提供大腦可讀的預期順序。實際 runtime 仍可由 edges 計算 DAG。

12

Capability Inventory：這是防止 LLM 幻覺的核心。

大腦只能根據目前能力設計 workflow；缺少的能力要被放進 missingCapabilities，而不是假裝存在。

{ "availableNodeTypes": [ "input.text", "model.llm", "model.image", "output.text" ], "availableCompilers": [ "compiler.none", "compiler.prompt", "compiler.file" ], "availableArtifactPolicies": [ "artifact.image.generated", "artifact.file.uploaded" ], "notAvailableYet": [ "model.video", "controller.ifElse", "brain.autoApply", "custom.codeSandbox" ] }

大腦輸出規則

如果想用 unavailable capability，只能出現在 proposal 的 `missingCapabilities[]`，不能出現在 executable nodes。這一條會讓 Codex 的下一輪任務變得精準。

13

編譯層：每個輸入都經過可插拔 compiler，沒有轉譯需求也走空編譯。

這對未來圖片、影片、文件、音訊、3D、表格都很重要。

compiler.none

不需要轉譯的文字、簡單參數、已符合 packet schema 的資料，仍然通過空編譯層，保留統一 pipeline。

compiler.file

上傳檔案先被記錄 metadata、storage id、mime、size、hash，再決定是否需要 OCR、transcode、embedding、thumbnail。

compiler.media

圖片或影片可在未來加上 frame extraction、reference parsing、style extraction、duration mapping。

input asset → asset record → compiler route → packet normalization → message/workflow packaging → runtime node input
14

Graph UI：畫布只顯示拓撲，細節要進 Inspector。

目前 graph 是可以用的，但 V23 要讓它變成 workflow design cockpit。

Canvas

顯示節點、連線、run state、validation badge、artifact badge、missing capability warning。不要把所有設定塞在卡片上。

Inspector

點擊節點後顯示 purpose、rationale、config、io、compiler、limits、run history、artifact policy。

Capability Palette

新增節點按鈕應該 schema-driven：可用節點能拖出，不可用節點灰階顯示原因與開發建議。

Brain Panel

顯示 summary、topology explanation、warnings、optimization proposal、diff、missingCapabilities。

Validation Overlay

將 validator 結果直接映射到節點與 edge，而不是只在 console 或 toast 顯示。

Proposal Diff

讓使用者看見大腦改了哪些節點、prompt、模型、edge、artifact policy。

15

Import / Export 生命週期：不要只有下載，要有驗證、預覽、套用、回滾。

一鍵導入 workflow 只有在可驗證與可回復的情況下才安全。

  • Export Workspace：完整備份，不改掉現有邏輯。
  • Export Workflow Contract：只輸出 `nexus.workflow.v1`，適合交給 LLM 或其他 NEXUS 導入。
  • Import Contract：先 parse，再 schema validate，再 capability validate，再顯示 preview。
  • Apply Contract：產生 runtimeLite graph 或更新現有 graph。
  • Rollback：導入前自動做 workspace snapshot，出錯能回復。
import flow: file selected → parse JSON → schema validate → capability validate → graph preview → snapshot current workspace → apply to graph/runtimeLite → save contract version
16

後端記錄策略：先不要重構所有表，先讓 contract 有版本身份。

Supabase 可以保存 JSONB，但不要濫用 JSONB 取代關聯資料與權限邊界。

資料建議儲存第一版優先級
Workflow Contract`workflow_contracts.contract_json jsonb` + version metadata高
Validation Result`workflow_contract_validations` 或 contract metadata高
Brain Proposal`workflow_proposals.proposal_json jsonb`中
Run Evidence沿用 runtime runs，後續加 checkpoint/event table中
Generated Artifacts沿用 artifact vault/storage metadata高

RLS 原則

workflow contract、proposal、artifact 都是 user/workspace-scoped。前端只能用 publishable/anon key 走 RLS；server-only 能力才可碰 service role，且不能外洩。

17

生成資產區：圖片、影片、音訊都要走同一套 artifact policy。

你要求下載、保存、歷史作品，這應該是一個跨模型能力，而不是 image node 專屬功能。

artifact record

保存 id、type、mime、storage path、url、thumbnail、source node、run id、model settings、prompt hash。

history dropdown

右上工具列的 generated history 應依 workspace/thread/workflow 篩選，並支援下載。

message rendering

聊天室圖片不應退化成醜預覽。message 應支援 text/image/video/file artifact block。

artifactPolicy: persist: true downloadable: true showInChat: true showInGeneratedHistory: true storageScope: workspace retainMetadata: model + prompt + quality + aspect + seed
18

RuntimeLite Bridge：contract 不直接執行，先 materialize 成 runtime plan。

這樣 V23 可以成長，但不破壞 V22 已經通過的 runtime 行為。

toRuntimeLite(contract): assert schema validated map contract.nodes to runtime nodes map contract.edges to runtime edges map execution.startNodes map artifactPolicy attach abort signals attach run metadata return RuntimeLiteStatePatch
fromRuntimeLite(runtime): inspect nodes/edges infer minimal workflow contract mark unknown rationale as "needs-human-label" attach existing graph positions preserve runtime ids where safe return draft contract

落地順序

先做 one-way `to-runtime-lite`，再做 `from-runtime-lite`。反向推論很有價值，但第一版不能讓它阻塞 contract validation。

19

Brain Context Pack：大腦讀的不是 raw database，而是整理好的上下文包。

這是讓「大腦像神經一樣感知目前節點狀況」的安全做法。

Design Context

contract、workflow intent、nodes、edges、rationale、capability inventory、success criteria。

Runtime Context

run id、node states、duration、errors、abort status、retry status、latest packets。

Artifact Context

生成資產、下載狀態、storage ids、metadata、模型設定、prompt/seed/quality/aspect。

WorkflowBrainContextPack: contract: nexus.workflow.v1 capabilities: current capability inventory runtimeEvidence: latest runs + node states artifacts: generated assets + uploaded inputs userGoalDelta: what Sean wants changed now guardrails: unavailable capabilities + security constraints
20

Brain Proposal：大腦不直接改系統，它輸出可審核的升級草案。

這是把 LLM 的創造力變成工程可控性的關鍵。

{ "schema": "nexus.workflow.proposal.v1", "summary": "Improve image generation workflow with prompt refinement and artifact policy.", "optimizedWorkflow": {}, "diff": [ { "op": "addNode", "nodeId": "prompt_refiner" }, { "op": "updateNodeConfig", "nodeId": "image_model_1" } ], "missingCapabilities": [ { "id": "controller.ifElse", "whyNeeded": "Route failed image prompts into revision loop.", "suggestedCodexTask": "Build conditional controller node." } ], "risks": [], "applyMode": "human-review-required" }

Proposal 的價值

它讓大腦可以大膽設計，但每個改動都被具體化。Codex 可以直接從 `missingCapabilities[].suggestedCodexTask` 取得下一輪工程任務。

21

Validator Ladder：從 JSON 正確到可安全執行，至少要五層檢查。

V23 的保護力來自 validator，不是來自相信 LLM 會乖。

層級檢查內容失敗結果
SyntaxJSON parse、schemaVersion、required root fields。不能導入。
Schemanodes/edges/io/config/artifactPolicy 欄位型別。不能預覽。
Topologyedge node ids、handles、duplicate ids、DAG、start/output。可預覽但不可執行。
Capabilitynode types、models、compilers、artifact policies 是否存在。列為 missing capability。
SecurityAPI keys、storage scope、user/workspace ownership、custom code 限制。阻擋並提示原因。
22

安全與權限：大腦可以看能力，但不能拿到不該拿的權限。

圖片、影片、檔案、workflow 都會接後端記錄，權限邊界必須從 V23 一開始就寫清楚。

API Key

前端不能接觸 OpenAI key 或 service role。圖片生成 route 在 server 執行，metadata 可以回傳，secret 不可回傳。

RLS

workflow contract、artifact、proposal 都應該 user/workspace scoped。匿名 key 可用但必須依賴 RLS。

Storage

資產上傳與生成檔案應走 storage API，不直接改 storage schema。metadata 和 object 路徑要分清。

Brain Access

Brain Context Pack 應該是最小必要資訊，不要把原始私密資料或 secrets 全部丟給模型。

Apply Gate

brain proposal 第一版 human-review-required。未來自動套用也應只針對低風險變更。

Audit

誰匯入、誰套用、哪個 proposal、哪個 run 產生哪些 artifact，都應可追蹤。

23

Run Evidence：大腦要能用執行證據修正 workflow。

這一層參考 checkpoint/super-step 思路，但不急著換掉 runtimeLite。

  • Run summary：每次 workflow start 的起點、時間、使用者、contract version。
  • Node evidence：每個節點的 status、duration、input packet、output packet、error。
  • Artifact evidence：模型設定、生成結果、storage record、download record。
  • Checkpoint idea：先記錄階段性狀態，未來才做 durable resume/time travel。

為什麼重要

如果大腦只讀設計，不看執行證據，它只能做理論優化。讀到 run evidence 之後，它才能說：哪個 prompt 常失敗、哪個模型慢、哪個 artifact 沒保存、哪條 edge packet 不穩。

24

Media Nodes：圖片、影片、音訊節點要共用同一套模式，而不是各做各的。

你前面要求 image API、品質、比例、下載、歷史紀錄，這正好是 media model node family 的起點。

節點家族必要設定共用輸出V23 狀態
model.imagemodel、quality、aspectRatio、count、prompt、reference assetsartifact.image先做標準化

UI 決策

圖片節點和未來影片節點都要有 download/save/history。右上 generated history 不應只認圖片，而應認 artifact type。

控制腦與 If/Else：要先有 packet schema，才能做可靠條件分支。

智能主控大腦不是魔法節點，它需要可讀的狀態、規則、限制與提案格式。

controller.ifElse

讀入 packet，根據規則或 LLM 判斷走不同 edge。第一版先做 rule-based，LLM decision 後做。

controller.merge

等待多個 parallel branches 完成，合併成 summary packet 或 artifact collection。

brain.supervisor

讀 context pack，提出 proposal。第一版不直接控制 runtime，避免不可預測。

safe controller ladder: 1. deterministic merge 2. rule-based if/else 3. validator-approved LLM decision 4. supervised proposal 5. limited auto-apply for low-risk changes

React Flow

風險與撞牆處理：高概念的敵人是範圍膨脹。

這頁把最可能拖垮 V23 的地方先寫成 stop/go criteria。

Schema 膨脹

如果欄位超過 validator 能覆蓋的範圍，立刻切回 strict core。未來能力放 `notAvailableYet[]`。

Runtime 衝突

如果 bridge 需要改大量 runtimeLite，停止，重新設計 adapter。V23 不應破壞 V22 執行基線。

後端過早

如果 contract 還不穩，不做 migration。先 local JSON + validator + fixtures。

UI 過載

Canvas 不承擔所有細節。節點卡片只露狀態和摘要，細節進 inspector。

LLM 幻覺

未註冊 capability 只能進 missingCapabilities，不能進 executable nodes。

安全越界

任何 contract 或 proposal 不得包含 secrets。API key/server role 永遠 server-only。

30 頁最後收束成一個工程入口：先讓一份 JSON 被正確定義、正確驗證、正確轉成 runtime。

第一輪工程的完成定義

  • 新增 `src/lib/workflow-contract/types.ts`，定義 `NexusWorkflowV1`、node、edge、packet、artifact、brain proposal。
  • 新增 `schema.ts` 與 `validator.ts`，能拒絕壞 JSON、壞 edge、未知 node type、未知 compiler。
  • 新增 fixtures：最小文字 workflow、圖片 workflow、並行 workflow、missing capability workflow。
  • 新增 tests：valid contract pass、invalid ids fail、unknown capability fail、runtime bridge output stable。
  • 不做後端 migration、不做自動套用、不做任意 loop，先讓核心可信。
