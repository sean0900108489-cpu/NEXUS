大腦理解修正版

這裡已依照你的修正重寫：NEXUS 的大腦不是事後看 telemetry 的監控器，而是從一開始就讀懂整份 workflow JSON 的設計腦。

目前專案實際怎麼記錄

  • 左上角 Export 目前下載的是完整 workspace snapshot：agents、panels、settings、graph、runtimeLite、notebooks。
  • 後端 `/api/v1/workspaces/[workspaceId]/state` 存的是 cloud snapshot，並投影成 graph、settings、theme、agent、memory、tool_state、branch。
  • runtimeLite 目前是執行層：`input.text`、`model.llm`、`model.image`、`output.text`、edges、runs、ContextPacket。
  • 所以現在有「狀態快照」與「執行圖」，但還沒有真正給大腦讀的「workflow 設計合約」。

下一層該補的合約

需要新增 `nexus.workflow.v1`：一份 JSON 就能生成 workflow，並且明確寫出 intent、節點目的、連線理由、順序、串聯/並聯、能力清單、限制、artifact policy、compiler、brain proposal 格式。

Canonical flow: Workspace Snapshot = recovery RuntimeLite = execution nexus.workflow.v1 = brain-readable design contract Brain output: optimizedWorkflow + missingCapabilities + suggestedCodexTasks

它要先讀懂什麼

整個 workflow 的目標、節點角色、edge 傳什麼、為什麼這樣串、哪些是串聯、哪些是並聯。

它要知道邊界

目前平台有哪些 node type、model、compiler、artifact 能力；哪些功能還不存在，不能假裝已經能做。

它要能反向產出

一鍵輸出新版 workflow JSON，補好提示詞、模型設定、節點連線，並列出缺少功能讓 Codex 下一輪升級。

V23 指令規劃預測

這一輪沒有動產品 source，而是把你的新方向收斂成可以接著實作的高精度藍圖：一份 JSON 能生成、理解、驗證、優化整個 workflow。

新起點

  • V22 的 graph/runtime/artifact history 已能支撐實作，但還缺一層 brain-readable workflow contract。
  • 現有 Export 要保留為 workspace snapshot，不應硬改成大腦格式。
  • 下一步應新增 `nexus.workflow.v1`，再用 bridge 轉成 runtimeLite，而不是讓 runtimeLite 直接背負設計語言。
  • 大腦輸出要包含 optimizedWorkflow、diff summary、missingCapabilities、suggestedCodexTasks，讓 Codex 下一輪能精準升級。
